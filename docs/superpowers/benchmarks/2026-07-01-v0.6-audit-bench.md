# v0.6 Audit Benchmark — `validate_ds_compliance` vs. five separate calls

Deterministic mock-figma fixture, **160 nodes** (40 per kind: raw fill / default
name / off-grid spacing / failing contrast). Fixture size is chosen so the
composite's shared 200-item `MAX_RESULTS` budget does **not** truncate — an
apples-to-apples comparison, not an artifact of dropped findings (the test
asserts the composite emits no truncation `_hint` at this size).

Payload bytes = `JSON.stringify(safeSerialize(result)).length`; approx tokens =
bytes / 4. Wall-clock ms is indicative only (sub-millisecond, not asserted).
Source: [`audit-bench.test.ts`](../../../packages/bridge-plugin/src/operations/__tests__/audit-bench.test.ts).

## Headline: the win is round-trips and scans, not payload bytes

| Path | Payload bytes | ~Tokens | Round-trips | Page scans |
|------|--------------:|--------:|:-----------:|:----------:|
| `lint_styles` + `lint_detached` + `lint_naming` + `check_contrast` + `audit_spacing` (5 calls) | 17,509 | 4,377 | **5** | **5** |
| `validate_ds_compliance` (1 call) | 28,444 | 7,111 | **1** | **1** |
| Delta | **+62% larger payload** | — | **−4 round-trips** | **−4 scans** |

Per-op breakdown of the five separate calls: `lint_styles` 8,143 B ·
`lint_detached` 121 B · `lint_naming` 2,364 B · `check_contrast` 3,968 B ·
`audit_spacing` 2,913 B.

## Honest reading

- **The composite payload is *larger*, not smaller** (+62%). Its per-finding
  shape (`CheckFinding`: `check` + `detail` + `nodeType` + `meta` on every item)
  carries more metadata than the leaner single-purpose op payloads, and it
  returns findings from all five checks in one response. So on raw payload
  bytes alone, the composite does **not** win. An earlier draft cited a "35%
  reduction" — that was measured on an oversized (400-node) fixture where the
  composite truncated at 200 findings while the five separate ops (each with
  their own full 200 budget) did not. That number was a **truncation artifact**
  and has been discarded.
- **The genuine savings are structural:**
  - **Round-trips 5 → 1.** Four MCP request/response envelopes are eliminated.
    In practice each MCP tool call also carries protocol framing and, more
    importantly, an agent reasoning turn to decide and issue the next call —
    none of which is captured by this payload-only measurement. For a *full*
    audit this is the dominant token cost, and it is where the composite wins.
  - **Page scans 5 → 1.** One `figma.currentPage.findAll()` walk instead of
    five — a real compute/latency saving that grows with page size.

## When to use which

- **Use `validate_ds_compliance`** when you want a full compliance sweep — it is
  one round-trip and one scan, and returns a severity-bucketed report. On a
  large page where total findings exceed `MAX_RESULTS`, it emits a truncation
  `_hint` and `counts` still reports the true per-category totals.
- **Use a single-purpose op** (`lint_styles`, `audit_spacing`, …) when you only
  need one dimension — its payload is far smaller (e.g. `lint_detached` at
  121 B vs the composite's 28 KB), so a targeted question stays cheap.

## Caveat

This measures serialized payload only. It does **not** capture MCP round-trip /
envelope overhead or inter-call agent reasoning — the dominant real-world token
cost of issuing five separate tool calls. The composite's true token advantage
for a full audit is therefore **understated** here; the payload delta is a
conservative (and, on its own, unfavorable) lower bound.

## Follow-up (next loop candidate)

The composite's payload could be made competitive by compacting its output
shape — e.g. dropping the redundant `check` field (implied by the bucket),
shortening or omitting `detail` for machine consumers, and pruning `meta`. That
is a deliberate design change deferred to a future loop, not folded into this
release.
