# v0.6 — Compact `validate_ds_compliance` Output

**Status:** Approved design (brainstorming complete)
**Date:** 2026-07-01
**Branch:** `feat/v0.6-trustworthy-cheaper-audits` (amends open PR #38)
**Loop:** 2 (follows the v0.6 audit hardening that shipped in PR #38)

## Context

The `validate_ds_compliance` composite op landed in PR #38 (v0.6). A benchmark
found its response payload is ~62% **larger** than the five separate audit calls
it replaces (28,444 B vs 17,509 B for the same 200 findings on a 160-node
fixture). Its genuine win is structural — 5→1 MCP round-trips and 5→1 page scans
— but the payload-size regression forces an awkward caveat on the "cheaper
audits" story.

Two independent measurements agree on where the bytes go (per-field, across all
200 findings in the composite response):

| Field | ~Bytes (incl. key) | Share | Consumers found in repo |
|-------|-------------------:|------:|-------------------------|
| `detail` | 7,307 | 25.7% | none — pure human string |
| `meta` | 6,987 | 24.6% | yes (the real payload) |
| `nodeType` | 3,267 | 11.5% | none — decorative |
| `check` | 3,427 | 12.0% | disambiguates P2 only |
| `nodeName` | 2,787 | 9.8% | essential |
| `nodeId` | 2,850 | 10.0% | essential |

`detail` is fully reconstructable from `check` + `meta` (+ `nodeName`) in every
case, and `nodeType` has zero programmatic consumers anywhere in the repo
(source grep, `dist/` excluded).

Because v0.6 is still an **unmerged PR**, reshaping the composite's output now has
**zero backward-compat cost** — the compact form becomes its v0.6 debut. This
loop folds into PR #38; no version change.

## Goal

Make `validate_ds_compliance` no worse than the five separate calls on payload
bytes (in addition to its existing round-trip / scan win), by removing redundant
per-finding fields — without losing any information a consumer needs.

## Non-goals (out of scope this loop)

- Aggressive compaction: grouping findings by `check` as a key, short/1-char
  keys, or flattening `meta` to top-level fields. (Considered; deferred — the
  approved scope is "moderate.")
- Dropping `check` (it is the only discriminator inside the P2 bucket, which
  mixes `detached` and `spacing`).
- Changing the shared `CheckFinding` type, the pure check functions, or any
  standalone op's output shape.
- Grouping findings by `nodeId` to dedupe multi-finding nodes (a real future
  win on production files, but not measurable on the current fixture).

## Design

### 1. Compact the composite's finding projection

The composite currently pushes whole `CheckFinding` objects
(`{ nodeId, nodeName, nodeType, check, detail, meta? }`) into its `by_severity`
buckets. Change it to project each finding to a compact object:

```
{ nodeId, nodeName, check, meta? }
```

- **Drop `detail`** — reconstructable from `check` + `meta` + `nodeName`; no
  consumer reads it.
- **Drop `nodeType`** — no consumer reads it; the leaner standalone ops
  (`check_contrast`, `audit_spacing`) do not carry it either.
- **Keep `check`** — needed to disambiguate `detached` vs `spacing` within P2,
  and cheap relative to its usefulness.
- **Keep `meta`** — the only non-derivable payload (`ratio`, `font_size`,
  `hex`, `property`, `value`, `base_unit`, `parentName`). Omit the `meta` key
  entirely when a check produces none (e.g. `naming`), rather than emitting
  `meta: undefined`/`{}`.

The shared `CheckFinding` type and the pure check functions are unchanged — they
still produce `detail`/`nodeType` internally; the composite simply does not emit
them. Implementation: a small `toCompactFinding(f: CheckFinding)` helper inside
`validate.ts` applied where findings are pushed to buckets (and for the inline
contrast finding, construct the compact object directly rather than building a
`detail` string).

### 2. Add a top-level `legend`

So a human-facing consumer loses nothing, add a `legend` object to the result,
paid once (~250 B) instead of ~7.3 KB of per-item `detail`:

```
legend: {
  contrast: "text contrast below WCAG AA",
  style: "raw fill/stroke/text/effect (no style or variable)",
  detached: "frame likely a detached instance",
  spacing: "auto-layout value off the base-unit grid",
  naming: "default layer name"
}
```

`legend` is a static constant in `validate.ts`. It is always present (its size
is trivial and its presence keeps the response self-describing).

### 3. New result shape

```jsonc
{
  "total_nodes": 160,
  "counts": { "contrast": 40, "style": 80, "detached": 0, "spacing": 40, "naming": 40 },
  "by_severity": {
    "P0": [{ "nodeId": "t3", "nodeName": "T", "check": "contrast", "meta": { "ratio": 1, "font_size": 12 } }],
    "P1": [{ "nodeId": "r0", "nodeName": "raw", "check": "style", "meta": { "property": "fill", "hex": "#ff0000" } }],
    "P2": [{ "nodeId": "a2", "nodeName": "Row", "check": "spacing", "meta": { "property": "itemSpacing", "value": 9, "base_unit": 8 } }],
    "P3": [{ "nodeId": "f1", "nodeName": "Frame 1", "check": "naming" }]
  },
  "legend": { "contrast": "…", "style": "…", "detached": "…", "spacing": "…", "naming": "…" },
  "summary": "…",
  "_hint": "…"   // only when truncated (unchanged from PR #38)
}
```

The `counts`, `summary`, truncation `_hint`, severity bucketing, and shared
`MAX_RESULTS` budget are all unchanged.

### 4. Manifest `returns`

Update the `validate_ds_compliance` manifest `returns` string to describe the
compact finding shape and the `legend` field.

### 5. Benchmark fixture fix (in passing)

The benchmark's `buildNodes` never creates `INSTANCE` nodes, so `lint_detached`
and the composite's `detached` check both always find 0 — P2 is 100% spacing.
Add an `INSTANCE`-typed node plus a matching detached-look-alike `FRAME` to the
fixture so `detached` is actually exercised in both paths (keeps the comparison
representative). Adjust the fixture's node count / expected `counts` assertions
accordingly.

## Expected result

Dropping `detail` (~7.3 KB) + `nodeType` (~3.3 KB) ≈ −10.6 KB, plus a ~250 B
legend, lands the composite at ~17.3 KB vs the standalone 17,509 B — **parity,
marginally under** — while retaining the 5→1 round-trip and 5→1 scan win. The
composite is then no worse than the separate calls on any axis, removing the
"payload is larger" caveat.

## Testing

- **Shape test** (`validate-ds.test.ts`): a composite finding has exactly
  `{ nodeId, nodeName, check }` plus `meta` when applicable, and **no** `detail`
  or `nodeType` key. `legend` is present with all five check kinds. Existing
  `counts`, bucket-length, and truncation-`_hint` assertions still hold.
- **Naming-finding test**: a `naming` finding omits `meta` entirely (no
  `meta: undefined`).
- **Benchmark** (`audit-bench.test.ts`): with the detached node added, assert
  `counts.detached > 0`. Re-measure; if the composite payload is now
  `<= sepBytes`, re-add `expect(composite.bytes).toBeLessThanOrEqual(sepBytes)`
  as a guard (it was dropped in PR #38 when the composite was larger). Update the
  emitted report + the benchmark doc with the new honest numbers.
