# PluginOS 0.6.0 — Trustworthy + Cheaper Audits

**Status:** Approved design (brainstorming complete)
**Date:** 2026-07-01
**Branch:** `feat/v0.6-trustworthy-cheaper-audits`

## Context

PluginOS exposes 26 Figma operations through 5 MCP tools. Its core value
proposition is **token efficiency** (~230 tokens/call via `safeSerialize`
capping). Two independent projects logged improvement notes while driving
PluginOS live:

- `Downloads/MCP Apps Lib research/BUGS-pluginos.md`
- `design-superpowers/docs/superpowers/improvements-pluginos.md`
- `design-superpowers/docs/superpowers/pluginos-improvements.md`

The feedback spans far more than one release. This spec covers **one coherent
loop**: making the audit suite *trustworthy* (it currently produces false
positives in any variables-based design system) and *cheaper* (bundling N lint
calls into one round-trip). Both directly serve the token value proposition.

### Constraint (non-negotiable)

Expand capability **without increasing token consumption**. This has two tax
vectors that the design must respect:

1. `list_operations` returns **full manifests** — every new op grows its output.
2. `packages/claude-plugin/skills/pluginos-figma/SKILL.md` has a **hard
   1150-token budget** (CI-enforced) — every new op pressures it.

The composite op in this release is **token-negative** (saves ~1150 tok/call at
runtime, costs one manifest + one SKILL line), so the net is favorable.

### Relevant architecture (verified)

- Operations self-register via `registerOperation({ manifest, execute })` in
  `packages/bridge-plugin/src/operations/*.ts`.
- `OperationContext` (in `operations/context.ts`) lazily exposes `nodes`
  (`figma.currentPage.selection` or `.findAll()`), `figma`, `params`,
  `MAX_RESULTS`.
- Scope reads `figma.currentPage` directly — **no page targeting exists**.
- Variable-blindness confirmed at source: `lint_styles`,
  `find_non_style_colors`, `audit_spacing`, `check_contrast`,
  `analyze_overrides`, `extract_palette` read only `*StyleId`/raw values.
  `boundVariables` is used **only** in prelude write-helpers, never in
  read/audit ops.
- Tests use Vitest with a mocked `figma` global, calling `handler.execute(ctx)`
  directly (`operations/__tests__/`).

## Goals

1. Audit ops stop flagging variable-bound properties as violations.
2. A single composite op returns a full compliance report in one round-trip.
3. Page-scoped ops can target a page explicitly and warn on likely-wrong pages.
4. `audit_spacing` actually flags off-grid values.

## Non-goals (explicitly deferred to later loops)

- "should-tokenize" advisory (raw value matching an existing variable).
- Library analytics / `count_instances_by_component` (multi-page traversal).
- New primitives (`get_selection_context`, `get_node_properties`,
  `apply_variables`, `find_overrides`).
- `execute_figma` timeout/rollback changes.
- `export_tokens` alias resolution; `extract_css` fixes.
- Recipe/doc fixes (SECTION relative coords, variant re-query, async-page).

## Design

### Work item 1 — Variable-awareness

**Problem:** On a clean screen with fills bound to semantic variables,
`find_non_style_colors` reported 20 "hardcoded colors" and `lint_styles`
reported 26 "Fill without style" — flagging correctly-bound nodes identically
to raw-hex drift. This makes the audit unable to separate good from bad.

**Fix:** Introduce a shared binding resolver:

```
resolveBindingState(node, property): "style" | "variable" | "raw"
```

- `style` — the corresponding `*StyleId` is a non-empty string.
- `variable` — the property is variable-bound. For fills/strokes this is
  **per-paint**: a paint is bound when `paint.boundVariables?.color` exists.
  For node-level fields (spacing, etc.) check `node.boundVariables?.[field]`.
- `raw` — neither.

A property is a **violation only when `raw`**. For array-valued paints
(`fills`, `strokes`), a violation requires at least one visible `SOLID` paint
that is `raw` (not covered by style or per-paint variable binding).

Reads must be defensive: `boundVariables` may be `undefined`; paint arrays may
be `figma.mixed`.

**Affected ops:** `lint_styles`, `find_non_style_colors`.

- Each issue gains a `binding` field (`"raw"` for emitted issues).
- Summaries report counts per state, e.g.
  `"Scanned N nodes. 3 raw, 12 variable-bound, 5 style-bound."`
- **Additive only** — no existing fields removed; ops that previously returned
  `issue: "Fill without style"` still do so for `raw` fills.

### Work item 2 — `validate_ds_compliance` composite

**One scan, five checks, one round-trip.** The op walks the resolved node set
**once** and runs all five checks per node:

1. variable-aware style lint (item 1 logic)
2. detached components (`lint_detached` logic)
3. naming (`lint_naming` logic)
4. contrast (`check_contrast` logic — text node vs resolved background)
5. spacing-grid (item 4 logic)

**Return shape:**

```jsonc
{
  "total_nodes": 240,
  "by_severity": {
    "P0": [{ "check": "contrast", "nodeId": "...", "nodeName": "...", "detail": "..." }],
    "P1": [...],
    "P2": [...],
    "P3": [...]
  },
  "counts": { "contrast": 2, "style": 5, "detached": 1, "spacing": 8, "naming": 3 },
  "summary": "..."
}
```

Findings are capped by `MAX_RESULTS` across the combined set (severity order
preserved: P0 first).

**Severity mapping (approved):**

| Bucket | Findings |
|--------|----------|
| **P0** | Contrast below WCAG AA (hard accessibility fail) |
| **P1** | Raw colors / fills with no style *and* no variable (drift) |
| **P2** | Detached instances · off-grid spacing |
| **P3** | Naming violations |

**Params:** `scope`, `confirm` (500-node guard, op is added to `GUARDED_OPS`),
`base_unit` (default 8, feeds check 5), and page-targeting params (item 3).
Optional `checks?: string[]` is **out of scope** for this release — always runs
all five to keep the manifest minimal.

**Token math:** replaces ~5 × ~230 = ~1150 tok with a single ~300 tok call.
Net cost to discovery: one manifest + one SKILL line.

### Work item 3 — Page targeting + warn-on-empty

**Problem:** `scope:"page"` silently follows `figma.currentPage`. A lint
sequence returned 0 findings (false-clean) because the current page had drifted
to an empty page.

**Fix:**

- Add optional `page_name` / `page_id` to page-scoped ops and the composite.
- When present, resolve the target `PageNode` via `loadAllPagesAsync()` and scan
  **that page's `.findAll()` directly — without `setCurrentPageAsync`** (the
  user's viewport does not move).
- Resolution lives in the context factory, which gains an **async
  pre-resolution path** when page-targeting params are present (the current
  `nodes` getter is sync; page targeting requires awaiting the page load before
  the op body runs).
- **Warn-on-empty:** when `scope:"page"` resolves to **0 candidate nodes**,
  attach a `_hint`: `"Page '<name>' has 0 matching nodes — likely the wrong
  page. Pass page_name to target explicitly."` — instead of reporting a clean
  pass.
- **Page-not-found:** return `{ error: "Page '<x>' not found",
  available_pages: [...] }` (a value, not a throw).

### Work item 4 — `audit_spacing` base_unit grid

**Problem:** On a drifted screen it listed `[0,8,9,12,13,16,17,20,29,32]` but
reported **0 violations** — no notion of a base unit.

**Fix:** Add `base_unit` param (default **8**). Flag any spacing value
(`itemSpacing`, `paddingLeft/Right/Top/Bottom`, `counterAxisSpacing`) that is
not a multiple of `base_unit`. `0` is a valid multiple. The same
`checkSpacing(node, baseUnit)` function feeds the composite.

## Architecture / refactor

New directory `packages/bridge-plugin/src/operations/checks/` with **pure
per-node check functions**:

```
checks/binding.ts   → resolveBindingState(node, property)
checks/style.ts     → checkStyleBinding(node): Finding[]
checks/detached.ts  → checkDetached(node): Finding[]
checks/naming.ts    → checkNaming(node): Finding[]
checks/contrast.ts  → checkContrast(node): Finding[]
checks/spacing.ts   → checkSpacing(node, baseUnit): Finding[]
```

Each function is pure (node in, findings out), independently testable, and
consumed by **both** the standalone op (thin wrapper) and the composite (single
loop). This removes duplication and keeps each unit small enough to reason about
in isolation. Existing ops are refactored to call these — behavior preserved
except for the variable-awareness change and the spacing grid.

## Data flow

`run_operation` → `code.ts` → `createOperationContext` (optional async page
resolution) → `op.execute(ctx)` → check functions → `withHint(result)` →
`safeSerialize` → MCP response.

## Error handling

- Page-not-found → `{ error, available_pages }` value, not a throw.
- Variable-binding reads defensive against `undefined` / `figma.mixed`.
- Composite: a single failing check must not abort the others (wrap per-check).

## Testing (Vitest + mock-figma pattern)

- **binding:** variable-bound fill (`paint.boundVariables.color` set) is **not**
  flagged by `lint_styles` or `find_non_style_colors`; raw hex still is;
  style-bound still passes.
- **composite:** returns all five categories; correct severity buckets;
  `findAll` is called **once** (spy); respects `MAX_RESULTS` with P0 first.
- **spacing:** `base_unit: 8` flags `9, 13, 17, 29`; passes `8, 16, 32, 0`.
- **page targeting:** resolves a named page's nodes without changing
  `currentPage`; emits warn-on-empty hint on a 0-node page; page-not-found
  returns the error value with `available_pages`.

## Benchmark (release exit criterion)

Before declaring the loop done, produce a **before/after performance comparison**
and record it in the repo. Two dimensions matter, both tied to the value prop:

- **Token cost** — measure the serialized response size (proxy: JSON byte / token
  count via the same path `safeSerialize` produces) for the 5 individual audit
  calls vs. the single `validate_ds_compliance` call over an identical node set.
  Expected: composite is materially cheaper than the sum of the five.
- **Wall-clock** — `_duration_ms` for the composite (single scan) vs. the sum of
  the five standalone ops (five scans) over the same node set.

Method: a reproducible bench script (checked in under `scripts/` or a test) that
runs against a fixed synthetic node fixture (mocked `figma`, same pattern as the
unit tests) so results are deterministic and CI-runnable — no live Figma needed.
Capture the numbers in the plan's final report.
